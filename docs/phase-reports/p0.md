## 2. Rubric (v1)

### 2.1 Design decisions — skills_coverage (owner-authored, 2026-07-13)

1. **Evidence/keyword split:** 5-band requires hands-on evidence (skill used in a project/work
   entry); 3-band accepts keyword-level (skills-list mention). High band keeps verification
   power; low band stays objective.
2. **Preferred skills never move the band (stance A):** screening checks the floor; fewer
   judgment points also serves pass^k stability.
3. **Adjacent skills cap at mid-band:** adjacency (e.g. PostgreSQL for MySQL) is
   transferable-capability evidence, not direct satisfaction — supports 3, never 5. (Rationale
   on record is capability-transfer, NOT "experimental convenience".)
4. **Dimension boundary — years:** skill-attached years requirements ("5+ years Java") are
   scored under experience_level, not skills_coverage.
5. **Dimension boundary — degrees:** education requirements go to education_domain_fit.
6. **Bundled requirements:** discrete three-value determination (covered / partial / absent);
   majority-absent triggers the low band; partial caps the band at 2. Not a ratio.
7. **Whole-dimension banding (Plan A);** per-item aggregation (Plan B) deferred to P2 as a
   contrast-rubric experiment. **Soft veto** (hard_requirements unmet → aggregate capped +
   force-gated) lives at the aggregation layer; cap value is P1 gate config.

### 2.1b Design decisions — remaining dimensions (owner-authored, 2026-07-14)

- **experience_level — proximity semantics:** evidenced years only (dated, role-matching
  segments; Summary self-claims excluded); pass/fail duty lives in hard_requirements — this
  dimension scores *how close*, so a veto-firing gap can still earn a 3 when the experience is
  real.
- **hard_requirements — weight-0 ledger:** re-reads every must/required item reusing other
  dimensions' determinations; met / indeterminate / unmet ↔ 5 / 3 / 0; never in the weighted
  mean. Veto wiring: 0 → cap aggregate + force gate; 3 → force gate only; 5 → no action. Band 3
  is "cannot decide", never partial credit.
- **education_domain_fit — occupation-relative:** fit judged against the JD's occupation, not a
  software-centric yardstick; industry verticals excluded (option A); bootcamps are not degrees
  here (their project evidence stays under skills_coverage).
- **Weights 0.5 / 0.3 / 0.2** (skills / experience / education) — **ratified by owner
  2026-07-15** (PR #3 closing rulings). Rationale on record: "Weights follow the corpus's own
  emphasis: skills lists dominate JD body text (0.5); 74% of JDs state a years requirement
  (0.3); 63% mention degrees (0.2). The employer's ink allocation is the weighting evidence.
  Revisable if labeling reveals a mismatch — weight revision would be a logged rubric change."
  (Percentages here are the reproduced values from `data/profile_jd_requirements.py` — see
  traceability below; the ruling's draft said 64% for degrees from the unrecorded original
  scan.)
- **Traceability (resolved 2026-07-14; completed 2026-07-15):** the corpus numbers behind the
  education design ("accounting 108/280, largest bucket; software 65, under a quarter; industry
  signal absent in 72/280") are generated by `data/profile_jd_domains.py` (deterministic over
  the pinned train.csv) and archived as finding
  [001](../findings/001-jd-corpus-is-cross-occupation-not-software.md). Re-ran 2026-07-14 and
  again 2026-07-15: every number reproduced. The JD requirement-pattern scan backing the weight
  rationale is now recorded as `data/profile_jd_requirements.py` (run 2026-07-15): must/required
  language 220/280 = 79%, years requirement 206/280 = 74%, degree mention 177/280 = 63%. The
  earlier ad-hoc scan quoted 77/74/64 but its command was never recorded; those numbers are
  retired in favor of the scripted, regenerable ones (pattern definitions documented in the
  script docstring). The 1–2 point deltas do not touch the weight ordering.

### 2.4 Corpus-scan method (occupation / industry — finding 001)

Procedure (`data/profile_jd_domains.py`, deterministic over the pinned train.csv at revision
`08978e2…`): collapse the 6,241 train rows to the 280 unique JD texts (first-occurrence order,
`dict.fromkeys`), lowercase, then match word-boundary regex keyword lists — nine occupational
buckets (each a fixed term list, e.g. accounting/fin = accountant, accounting, bookkeeper,
financial analyst, auditor) and 23 industry/sector terms. A JD counts toward a bucket if any of
the bucket's terms matches; industry prevalence = share of unique JDs matching ≥1 industry term.

Reproduced counts (re-run 2026-07-15, all values match finding 001): accounting/fin 108,
software eng 65, sales/mktg 37, business/PM 36, data/ML 33, hr/admin 26, hardware eng 23,
devops/infra 17, healthcare 1; industry signal 208/280 (72 with none); top industry tokens
financial 94, finance 66, education 63, insurance 49, medical 45.

Confound warning (verbatim from finding 001): "keyword matching is coarse, and cross-axis terms
confound the two counts — `financial` matches both the finance *occupation* ('financial
analyst') and the finance *industry* ('financial services'), so the occupational
'accounting/finance 108' and the industry 'financial 94' overlap. Counts are directional
signposts, not exact partitions; precise separation would require manual coding of a JD
sample."

## 5b. Disagreement classification

A disagreement is **not an error to be resolved (eliminated)** but a signal to be
**classified** so it testifies in the right chapter. Each class carries a *disposition* — a
bookkeeping destination, not a repair. Discriminant, applied in order:

1. **rubric-gap** — the rubric does not cover the situation → *disposition:* rubric changelog
   candidate (a future revision, if the pain recurs — D4).
2. **ambiguous-label** — rubric clear, but both readings are defensible → *disposition:*
   labeling protocol's known limitations; a second-annotator signal.
3. **agent-error** (incl. *model-limitation* subclass — rubric clear, reference correct, model
   read wrong for a stable reason) → *disposition:* counts against the agent in P2 agreement.

Confidence marker per row: **certain** · **needs-mentor** (a human judgment call to batch to
touchpoint) · **needs-P2-data** (terminal call needs the trajectory's evidence, deferred to
P2's disagreement chapter). This table is **first-pass**; it exists so P2 inherits a populated
ledger instead of re-reading 30 trajectories.

## 5d. Disposition (not resolution)

No score is adjudicated to a single "correct" value and the rubric is not revised now (D4 —
between-rater variance on the adjacency axis is a *measured quantity*). Each class is routed
per its disposition above; the standing outputs: (1) this is the project's first disagreement
dataset and it localizes the rubric's least-specified axis — adjacency/occupation-boundary
strictness — as the concrete prediction for where P2's agent-vs-reference agreement is
weakest (**education_domain_fit** already the lowest human-human agreement at 8/10, and the
dominant agent-vs-reference pattern); (2) P2 reports per-dimension agreement stratified so this
axis stays visible, not averaged away (aligns with finding 009); (3) P2 inherits tables A/B
as its disagreement-analysis starting ledger — no trajectory needs re-reading to know where to
look.
