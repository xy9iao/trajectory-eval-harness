## 2. Rubric (v1)

### 2.1 Design decisions — skills_coverage (owner-authored, 2026-07-13)

1. **Evidence/keyword split:** 5-band requires hands-on evidence (skill used in a project/work
   entry); 3-band accepts keyword-level (skills-list mention). High band keeps verification
   power; low band stays objective.
2. **Preferred skills never move the band (stance A):** screening checks the floor; fewer
   judgment points also serves pass^k stability.
3. **Adjacent skills cap at mid-band:** adjacency (e.g. PostgreSQL for MySQL) is
   transferable-capability evidence, not direct satisfaction — supports 3, never 5. (Rationale
   on record is capability-transfer, NOT "experimental convenience".)
4. **Dimension boundary — years:** skill-attached years requirements ("5+ years Java") are
   scored under experience_level, not skills_coverage.
5. **Dimension boundary — degrees:** education requirements go to education_domain_fit.
6. **Bundled requirements:** discrete three-value determination (covered / partial / absent);
   majority-absent triggers the low band; partial locks out the 5-band. Not a ratio.
7. **Whole-dimension banding (Plan A);** per-item aggregation (Plan B) deferred to P2 as a
   contrast-rubric experiment. **Soft veto** (hard_requirements unmet → aggregate capped +
   force-gated) lives at the aggregation layer; cap value is P1 gate config.

### 2.1b Design decisions — remaining dimensions (owner-authored, 2026-07-14)

- **experience_level — proximity semantics:** evidenced years only (dated, role-matching
  segments; Summary self-claims excluded); pass/fail duty lives in hard_requirements — this
  dimension scores *how close*, so a veto-firing gap can still earn a 3 when the experience is
  real.
- **hard_requirements — weight-0 ledger:** re-reads every must/required item reusing other
  dimensions' determinations; met / indeterminate / unmet ↔ 5 / 3 / 0; never in the weighted
  mean. Veto wiring: 0 → cap aggregate + force gate; 3 → force gate only; 5 → no action. Band 3
  is "cannot decide", never partial credit.
- **education_domain_fit — occupation-relative:** fit judged against the JD's occupation, not a
  software-centric yardstick; industry verticals excluded (option A); bootcamps are not degrees
  here (their project evidence stays under skills_coverage).
- **Weights 0.5 / 0.3 / 0.2** (skills / experience / education) — `[owner ratification + one-line
  rationale pending]`.
- **Traceability (resolved 2026-07-14):** the corpus numbers behind the education design
  ("accounting 108/280, largest bucket; software 65, under a quarter; industry signal absent in
  72/280") are generated by `data/profile_jd_domains.py` (deterministic over the pinned
  train.csv) and archived as finding
  [001](../findings/001-jd-corpus-is-cross-occupation-not-software.md). CC re-ran the script on
  2026-07-14: every number reproduced. The JD requirement-pattern scan backing the weight
  rationale (77% must-language, 74% years, 64% degrees) still gets its command recorded here at
  closure.
